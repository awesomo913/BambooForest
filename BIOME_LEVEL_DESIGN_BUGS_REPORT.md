## CRITICAL (Game-Breaking / Softlock / Unplayable)

### C1. Level 18: Reverse Gravity Ceiling Landing is Broken — Player Can Never Stand on Ceilings

- **Files:** `sprites.py:808-820`, `levels.py:1067-1069`
- **Bug:** The Y-collision code in `Player.update()` only sets `is_on_ground = True` when the player lands on TOP of a platform (`dy > 0` branch, line 808). In reverse gravity, `velocity_y` becomes increasingly negative (player falls *upward*), so `dy` is always negative, routing to the "bonked head" branch (line 817-819). The player is repeatedly shoved below the ceiling platform by `self.rect.top = hit.rect.bottom`, then pulled back up by reverse gravity next frame, creating a continuous bounce — jumps_remaining is never refreshed.
- **Impact:** The ceiling platforms at y=180 (levels.py:1067-1069) are unreachable as standing surfaces. Reverse-gravity tunnel crossing is impossible.
- **Proposed Fix:** Add a reverse-gravity branch in the Y-collision handler (sprites.py:808-820). When `gravity_multiplier < 0`, treat landing on the UNDERSIDE of a platform as ground contact — set `is_on_ground = True`, refresh jumps_remaining, and snap `self.rect.top = hit.rect.bottom`.

### C2. Level 6: Wind Zone Pushes Player Rect Directly, Bypassing Velocity and Wall Collisions

- **File:** `game.py:491-495`
- **Bug:** Wind zones do `self.player.rect.x += math.floor(wz.get_push() * effective_dt)` — direct rect manipulation rather than adding to `player.velocity_x`. This happens AFTER `player.update()` has already resolved X collisions. A wind push can shove the player through walls, or worse, the wall-collision-reset (`self.velocity_x = 0` in `Player.update`) has already happened and can't counteract the wind push.
- **Impact:** Player can be pushed through solid walls by wind zones. Wall collision response is inconsistent with wind.
- **Proposed Fix:** In game.py:491-495, replace direct rect.x modification with `self.player.velocity_x += wz.get_push()`. Let the player's native collision resolver handle the resulting velocity. The wind velocity should be additive, not teleporting.

### C3. Level 4: AshBat Swoop Target is a Snapshot, Bat Can Be Kited Off-Screen

- **File:** `biomes.py:793-803`
- **Bug:** When AshBat transitions from "hover" to "swoop", it records `self.swoop_tx = float(player.rect.centerx)` and `self.swoop_ty = float(player.rect.centery)` as a one-time snapshot. It then flies to that exact position and returns to origin. The bat has no boundary clamping — if the player was moving right when the bat targeted, the bat flies to the snapshot x,y which could be off-screen.
- **Severity:** The bat never re-targets mid-swoop, so it's easy to bait-and-dodge. But it can end up stuck far from its origin if the player moves quickly.
- **Proposed Fix:** Clamp the swoop target to within reasonable distance of the bat's origin, OR add re-targeting every ~200ms during swoop.

### C4. Level 18: GravityDrone Uses Fragile Class-Name Check

- **File:** `game.py:617`
- **Bug:** `if enemy.__class__.__name__ == "GravityDrone"` — fragile string-based type checking. If the class is ever renamed, subclassed, or imported differently, this silently breaks.
- **Impact:** GravityDrone pull effect stops working.
- **Proposed Fix:** Either (a) use `isinstance()` with an import, or (b) give GravityDrone an `is_gravity_drone` attribute and check that.

### C5. Level 17: PhaseWraith Has `teleport_to()` Method But NEVER Calls It

- **File:** `biomes.py:1829-1864`
- **Bug:** `PhaseWraith` defines `teleport_to()` at line 1854 and has `teleport_cooldown: float = 0.0` (line 1827), but the `update()` method (lines 1829-1852) only does horizontal patrol + sine float. The `teleport_to()` method is **dead code** — never invoked from anywhere in the codebase.
- **Impact:** PhaseWraiths do not use portals despite being described as "Teleport — uses portals too" in the UI. This is a missing feature that significantly impacts the intended level 17 experience.
- **Proposed Fix:** In `game.py`'s portal update section (around line 582-604), add logic to check if a PhaseWraith is within range of an active portal and call `wraith.teleport_to(partner_x, partner_y)`.

### C6. Level 15: Rising Lava Kills Instantly, Bypasses Invincibility Frames

- **File:** `game.py:566-571`
- **Bug:** Rising lava contact directly sets `self.player.health = 0` and `self.player.dead = True` with NO check of `self.player.invincible_timer`. If the player somehow clips the lava surface while having i-frames (e.g., from a previous hit), they still die instantly.
- **Impact:** Inconsistent with every other damage source in the game which respects invincibility.
- **Proposed Fix:** Check `self.player.invincible_timer > 0` before the lava death check. Apply lethal damage (PLAYER_MAX_HP) instead of setting health=0 directly, so i-frames and damage events flow properly.

## MAJOR (Gameplay / Progression / Balance)

### M1. Level 6: Thermal Updraft Uses `max()` Instead of `min()` for Negative Force

- **File:** `game.py:499-502`
- **Bug:** Line 501: `self.player.velocity_y = max(self.player.velocity_y + THERMAL_FORCE * effective_dt, THERMAL_FORCE)`. `THERMAL_FORCE = -500.0`. If the player is falling fast (velocity_y = +700), then `velocity_y + (-500)*0.016 = ~692`, and `max(692, -500) = 692`. The updraft has negated its own effect because `max()` picks the larger (less negative) value. The intent is to clamp upward velocity to -500 (peak push), which requires `min()`.
- **Impact:** Updrafts barely work when the player is falling from height. A falling player receives negligible upward correction.
- **Proposed Fix:** Change `max()` to `min()` on line 501. `min(upward_push, -500)` correctly caps the upward velocity.

### M2. Level 4: Geyser Collision Rect Switches Between Dormant and Erupting — Inconsistent Hitbox

- **Files:** `biomes.py:489-505`, `game.py:458-465`
- **Bug:** The geyser has two rects: `_off_rect` (24px tall) and `_on_rect` (200px tall). The collision check `pygame.sprite.collide_rect(self.player, geyser)` uses the CURRENT rect. When dormant, the collision zone is a tiny 24px ellipse barely above the vent. The player can stand directly on the dormant geyser and the collision won't trigger. When erupting, the collision zone is a 200px tall column — the player can trigger the launch by walking into the eruption column from the side.
- **Impact:** Player can stand inside a dormant geyser vent and not get launched (false negative), or get launched by touching the eruption VFX (false positive). The intended mechanic is "stand in the vent opening when it erupts" — currently the dormant vent is impossible to stand in.
- **Proposed Fix:** Use a single consistent collision rect that covers the vent opening at all times (the dormant rect or a fixed 44x24 bottom area). Only check `is_active()` to decide whether the collision applies a launch.

### M3. Level 5: Crumbling Platform Detection Uses Wide Tolerance Zone

- **File:** `game.py:481-487`
- **Bug:** The crumbling detection creates a test rect 4px above and 2px wider than the crumbling platform: `test = pygame.Rect(cp.rect.x - 2, cp.rect.y - 4, cp.rect.w + 4, 8)`. This tolerance zone can trigger by the player jumping near the platform or head-bonking from below.
- **Impact:** Platforms can crumble without the player standing on them if they brush against the oversized detection zone.
- **Proposed Fix:** Use a tighter tolerance (1px wider, 1px above) and add a check that the player's `velocity_y >= 0` (not jumping up through the platform).

### M4. Level 8: BrineShard Grows Based on Player Velocity, But Ice Physics Makes Standing Still Hard

- **File:** `biomes.py:1195-1208`
- **Bug:** The BrineShard grows when `abs(player.velocity_x) < 10` (line 1201). On ice surfaces (friction_mode="ice"), the player slides and velocity_x bleeds off slowly even with no input. However, the `velocity_x *= ICE_FRICTION` (0.97) means it takes ~75 frames (1.25s) to drop below 10 from a modest 360px/s. Ice platforms create extended periods of near-still movement that triggers growth. Combined with the sliding, this is disproportionately punishing.
- **Impact:** Ice+brine sections are harder than intended — the BrineShard grows during the natural slide-deceleration on ice.
- **Proposed Fix:** Use a timer-based "standing still" check (standing_still_seconds > 0.3) instead of instantaneous velocity threshold, to account for ice drift.

### M5. Level 14: Mushroom Spring at x=3350 Reaches Crumbling Platform at y=280, But Crumbling Creates Follow-up Dead End

- **File:** `levels.py:882-888`
- **Bug:** The mushroom at (3350, FLOOR_Y=490) launches the player to the crumbling platform at (3350, 280) with peak height 336px (210px gap is jumpable). But after the crumbling platform crumbles, the next reachable platform is at (3100, 280) which is a horizontal gap of 250px from the mushroom. With no mushroom near 3100, the player can't reach that platform from the floor (280px vertical from FLOOR_Y). The intended path relies on perfect timing: bounce on 3350 mushroom, land on crumbling platform, jump across to 3100 before it crumbles.
- **Impact:** If the player bounces on 3350 mushroom but overshoots left past the crumbling platform, they fall back to FLOOR_Y with no way to reach 3100 — the crumbling platform is gone and no mushroom exists at x=2800-3100.
- **Proposed Fix:** Add a mushroom spring at x=2900-3000 to give the player a second bounce option after missing the crumbling platform.

### M6. Level 15: Lava Pause Y-sorting Has Redundant Pass, and First Pause is Below All Platforms

- **File:** `biomes.py:1441-1445`
- **Bug (cosmetic):** Line 1441 sorts ascending, line 1445 immediately re-sorts descending. Dead code on line 1441. Functionally: pause_ys become `[460, 410, 360, 310]`. The lowest platform is at y=430 (levels.py:904). Lava pauses at y=460, which is 30px BELOW the lowest platform. The platform at y=430 ("PlatformDef(400, 430, 240)") is reachable from the floor, but the player can stand on it while lava pauses at 460. When lava resumes and reaches y=430, that platform gets submerged.
- **Impact:** The lava's first pause provides no real gameplay benefit — the player is already clear of it. The second pause at 410 is more impactful.
- **Proposed Fix:** Remove line 1441 (dead sort). Optionally adjust first pause_y to y=420 so the lowest platform is threatened.

## MINOR (Quality of Life / Visual / Edge Cases)

### N1. Level 7: Dark Wall Added to Platforms Group Twice on Construction

- **File:** `biomes.py:2283, 2301`
- **Bug:** Line 2283: `platforms_group.add(self)` in `__init__`. Then in `update()` line 2301: `self._platforms.add(self)` when `should_solid`. Since the wall starts solid (line 2280), it gets added on init AND on the first update frame. `pygame.sprite.Group.add()` is idempotent, so this isn't a crash, but it's wasted work.
- **Impact:** None beyond redundant group operations.
- **Proposed Fix:** Remove the `init`-time add (line 2283), rely on update to manage group membership. Or add a guard `if self not in self._platforms`.

### N2. Level 16: TidalCrab Re-spawns at y=-40 After Falling Below y=600

- **File:** `biomes.py:1711-1714`
- **Bug:** When a tidal crab falls off its timed gate, lines 1712-1714 respawn it at y=-40. This creates a visual pop — the crab vanishes and reappears at the top of the screen, then falls back down. If the gate cycles back to solid while the crab is falling, the crab might land on a different gate than intended.
- **Impact:** Visual glitch + potentially confusing crab behavior.
- **Proposed Fix:** Instead of teleporting to -40, keep the crab falling and let it land on whatever solid surface is below (including the floor). If it falls off-screen, kill it.

### N3. Level 18: Wind Zone Overlaps Gravity Zone — Unpredictable Player Behavior

- **File:** `levels.py:1101-1110`
- **Bug:** Low-gravity zone spans (2300, 200, 500, 240). Wind zone spans (2600, 250, 150, 180). The overlap region from x=2600 to x=2800, y=250 to y=440 has both low gravity and wind push. Because wind pushes directly on rect.x (M2) and gravity affects velocity_y, the player experiences disjointed movement in the overlap.
- **Impact:** Confusing player experience in the overlap zone.
- **Proposed Fix:** Move the wind zone outside the gravity zone, or handle both through velocity additions.

### N4. Level 14: SporePuffer Flash Effect Re-renders Base Surface Every Frame During Flash

- **File:** `biomes.py:1400-1404`
- **Bug:** When `self.flash > 0`, the code does `img = self._base.copy()` followed by `img.fill((255, 255, 255, 80), special_flags=pygame.BLEND_RGBA_ADD)`. For a flash that typically lasts ~0.1s across 6 frames, this creates 6 new Surface copies. For 3+ SporePuffers on screen, this is ~18 extra surface copies per flash cycle.
- **Impact:** Minor performance hit during flash cycles. Same pattern repeated throughout biomes.py (MagmaLeaper, TidalCrab, PhaseWraith, HomingSpecter, VoidEater, etc.).
- **Proposed Fix:** Pre-render flash frames as class-level cache rather than per-update copies.

### N5. Level 9: Abyssal Trench Uses "cave" Biome with is_dark=True — Overlaps Level 7 Identically

- **File:** `levels.py:639-673`
- **Bug:** Level 9 uses `biome="cave", is_dark=True` and has crystals + dark walls — the exact same setup as Level 7 (Karst Caves). The visual and mechanical identity between these levels is nearly 100%. The only difference is enemy types (flying + chasers instead of spiders + glowworms) and platform layout.
- **Impact:** Level 9 feels like a re-skin of Level 7 rather than a distinct biome experience.
- **Proposed Fix:** Give Level 9 a unique biome (e.g., "abyss" with deep blue/black tile set) and consider a different dark mechanic (e.g., timed light bubbles instead of crystal striking).

## TINY (Code Quality / Comments / Typos)

### T1. Missing `checkpoint_positions` Parameter in Many Level Builders

- **File:** `levels.py` — levels 1 and 2 use `checkpoint_positions=[...]`;
  levels 4-5 use it; but it's absent from some builders...
  Actually all builders include it. No issue.

### T2. SulfurSlime Trails Cleared at Start of Each Update Frame

- **File:** `biomes.py:733`
- **Note:** `self._pending_trails.clear()` at the start of every update. Since trails are generated mid-update (line 751-753), this is correct — they're collected by `get_new_trails()` in game.py. No bug, just worth noting the design.

### T3. Level 15: ForgeHammer `die()` is a No-op But `alive()` Returns Hardcoded True

- **File:** `biomes.py:2122-2126`
- **Bug:** `die()` does nothing (line 2122: `pass`), and `alive()` returns `True` unconditionally (line 2125). The ForgeHammer can never be killed. This is presumably intentional (it's an indestructible hazard), but the `is_stompable: bool = False` already communicates that.
- **Impact:** If any future code calls `hammer.die()` expecting it to stop, the hammer remains active forever.
- **Proposed Fix:** Add `self.alive_flag = False; self.kill()` to `die()` for consistency, even if no invincible-override code currently calls it.

### T4. FalseGlowworm: `die()` is Also a No-op — Inaccessible

- **File:** `biomes.py:1170-1171`
- **Note:** `die()` does nothing here too. The glowworm is intentionally invincible (`is_stompable = False`) and `die()` is called from game.py's stomp/shuriken/ice logic. Since `is_stompable = False`, the stomp code never calls `die()`. The shuriken check (game.py:750) explicitly skips `BrineShard` and `DustDevil` but does NOT skip `FalseGlowworm` — so a shuriken WILL call `die()` on a FalseGlowworm, but since `die()` is a no-op, nothing happens. This is functionally correct but confusing.
- **Proposed Fix:** Make the no-op explicit with a comment, or add FalseGlowworm to the invincible skip list at game.py:750.

### T5. Level 10: Platform Gap (680, 760) is Only 80px — Hardly a "Trench"

- **File:** `levels.py:709`
- **Note:** Trenches defined as `(680, 760)`, `(1840, 1950)`, `(3380, 3490)`. The first gap is only 80px wide — easily jumpable but barely registers as a death pit. The player might not realize it's lethal.
- **Impact:** Minor UX confusion.

### T6. Level 12: Six Trenches Total — Most of Any Level

- **File:** `levels.py:793`
- **Note:** Level 12 "Tabletop Canopy" has 6 trenches, highest count of any level. Combined with 16 platforms and 2 moving platforms, this creates the most "porous" floor layout.
- **Impact:** No bug — design observation.

### T7. `_outro_anchor_x` Attribute Accessed with `hasattr` Guard

- **File:** `game.py:962, 970, 976`
- **Codestyle:** Lines 962 sets `self._outro_anchor_x` inside the `if self._outro_timer > 1.4` branch. Lines 970 and 976 use `if hasattr(self, '_outro_anchor_x'):` as a safety. Technically, `_outro_anchor_x` is guaranteed to be set on any frame after the first tick the outro is active (because the first tick always enters the >1.4 branch). But setting it via `hasattr` suggests uncertainty.
- **Impact:** None — just defensive coding.

## BOSS FIGHT SOFTLOCK ANALYSIS

### Level 3: Mutant Lair Boss Arena

- Arena: `PlatformDef(5100, 420, 700, 20)` (levels.py:447) — 700px wide platform
- Boss spawn: `(5500, FLOOR_Y)` — center of arena
- Goal zone: `goal_x=5850` — at the end of the arena
- **No walls or barriers at the arena edge** — the boss can knock the player off with non-stunned head-knockback (game.py:889-893: `velocity_x = 500.0 * kb_dir`). 500px/s knockback off a 700px platform gives ~1.4s flight time → player falls into trench. Trench death is handled by TRENCH_DEATH_Y check.
- **Softlock potential:** If the player falls off the arena platform repeatedly (losing all lives), they game over. This is intentional difficulty. However, if the boss knockback sends the player past the goal zone (x=5850) while the alarm timer is active, they could briefly touch the SafeZone, trigger "DEFEAT THE BOSS FIRST!" but be off the platform and die. **Grade: MINOR** — expected boss fight behavior.

### Level 13: Crystal Geode Boss Arena

- Arena: `PlatformDef(5100, 420, 900, 20)` (levels.py:814) — 900px wide arena
- Dark biome (`is_dark=True`) — player must hit crystals to see the arena boundaries
- Boss at `(5500, FLOOR_Y)`
- **Darkness + knockback softlock:** The arena is dark with only distant crystals (x=5400 is nearest). If the boss knocks the player off the arena platform in the dark, the player can't see where the arena ends. They might run in the wrong direction, miss the platform, and fall repeatedly.
- **Mitigation:** Checkpoint at x=5800 exists ("checkpoint_positions": [1800, 3500, 4900, 5800]). But 5800 is AT the arena — the player respawns on the arena. If they keep getting knocked off by the boss, they lose lives until game over.
- **Grade:** **MAJOR** — the dark arena + no visible boundaries + boss knockback is a compounding design issue.

### Level 18: Gravity Engine Final Boss Arena

- Arena: `PlatformDef(6700, 420, 700, 20)` (levels.py:1074) — 700px wide
- Boss at `(7000, FLOOR_Y)` — same pattern as L3/L13 bosses
- High-gravity zone nearby: `(5600, 300, 700, 200, "high")` — but this doesn't cover the arena (arena starts at 6700)
- **Boss knockback + high gravity:** If the player steps into the high-gravity zone before the arena, their jump height is halved. The boss could knock them back into the high-gravity zone where movement is harder. However, the arena itself is in normal gravity.
- **Grade: MINOR** — the gravity zone is far enough from the arena to not cause problems.

## LEVEL TRANSITION / SCORE / HEALTH CARRY ANALYSIS

### Carry Flow (game.py:344-358, 363-373)

1. `_advance_level()` stores `self._carry_score = self.player.score` and `self._carry_health = self.player.health`
2. `LevelTransition` runs for 2.0s (ui.py:836)
3. On completion, `_load_level(next_num)` creates fresh Player at `(100, FLOOR_Y)`
4. Then `self.player.score = self._carry_score` and `self.player.health = self._carry_health` are applied
5. Ice magic persists: `self._has_ice_magic_permanent` is checked and applied in `_load_level` (lines 275-277)

**Observations:**
- **T8. No bug found** — the carry chain is complete and consistent. Score and health survive level transitions correctly.
- The Player is created fresh then overwritten with carry values. This is slightly wasteful but correct.
- **Edge case:** If `_carry_health` is 0 (player barely survived), the new level starts with 0 HP. But the player is also `dead=False` until next damage check. The player gets one frame of being alive with 0 HP before dying. **Grade: TINY** — practically invisible.

## NPC INTERACTION ANALYSIS

### NPC Proximity Check (biomes.py:1281-1284)

- `self.show_dialog = dist < NPC_RANGE` where `NPC_RANGE = 60.0` (config.py:139)
- 60px is very small — the player must be nearly on top of the NPC to trigger dialog
- Dialog is drawn in `_draw_npc_textbox()` (game.py:1366-1391) — full-width, non-blocking

**Issues:**
- **N9. No NPC dialog cycling.** Each NPC has `dialog_lines` list but `current_line` is never incremented (biomes.py:1278). The dialog shows ALL lines at once. This is fine but the `current_line` field is dead code.
- **N10. No "press to interact".** Dialog fires automatically on proximity with no player input required. This is fine for a platformer but means the player can't re-read dialog.
